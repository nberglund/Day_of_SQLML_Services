Part 1 - Permissions

In the “real world” your users are probably not sysadmin’s or data base owners, so what happens when a user want to execute SPEES? So, open the permissions.sql in SSMS (log in as sa). You find the file here.

NOTE: The numbers -- 1. and so on you see in the script file corresponds to the numbered points below.

  1. Execute the script code and make sure all works.
  2. Impersonate the sqlsat1 user and see what happens when that user tries to execute the same code.

Oh dear me! That did not work as expected. Let us fix this. The way to fix it is to assign the correct permissions.

  1. Start by reverting back to sa.
  2. Check you can execute as sa.
  3. Assign the necessary permissions.

HINT: Permissions are needed both on sp_execute_external_script as well as execute on EXTERNAL SCRIPT. If you need help, the slide deck has some code.

  1. When you think you have fixed the permission issues, impersonate sqlsat1 and try again.
  2. When you are done, REVERT back to sa.

Part 2 - Scripts

In this part of the lab we work with @script parameter and your work-file is scripts.sql in the directory as above.

As sa open the file in SSMS and look at the at right underneath the USE SqlSatPreCon statement. There you see some code like so:

setosa <- iris[iris$Species == "setosa",]
meanSepWidth <- mean(setosa$Sepal.Width)
cat(paste("Seposa sepal mean width: ", meanSepWidth))

Code Snippet 1: R Script

Here we use the iris dataset which comes with SQL Server Machine Learning Services, and you see in Code Snippet 1 how we:

  • Filter on the Species label (we are interested in the “setosa” specie).
  • Calculate the mean of the sepal width.
  • Print out the mean value.

You are now, in the work-file, to:

  1. Assign the script to the @script parameter and execute.
  2. Execute as in 1, but use a script parameter instead.
  3. Take the script, put it into a file and load the file in the @script parameter, and execute. For this to work you need to:
  • Put the file somewhere where SQL Server can read it.
  • Use the R source function to load.

NOTE: In step 3 above, don;t forget R’s UNIX roots so when you define a path, you either define it in UNIX style: c:/r-scripts/iris_r.r, or by escaping \ like so: c:\\r-scripts\\iris_r.r.

Part 3 - Input & Output Data

In this, the last, part of this lab we load data from the database instead of from the external engine, and we push it into the script engine.

In SSMS open the dataset.sql file from here. You see how the top part of it looks almost the same as the one we used in Part 2 above. The one difference is that below the script is a SELECT statement: SELECT * FROM dbo.tb_irisdata_full. The table dbo.tb_irisdata_full has exactly the same data as the iris dataset, in fact we created the table in the setup from the iris dataset.

  1. Change the script to instead of using the iris dataset, use a dataset that you push in, but do the filtering still in the script file.

HINT: Use the @input_data_1 parameter and the InputDataSet variable inside the script.

  1. Now, change the script to not do any filtering, but filter in the data you push in.
  2. Let us work with data coming back out. If you select from the dbo.tb_irisdata_full table you see a column named SepalLength. We now want to return that column as a resultset coming back.

HINT: You use OutputDataSet and the way you reference a column in R is like so: InputDataSet$SepalLength. The last thing you need to think about is that OutputDataSet expects a data.frame.

Notice that when the resultset comes back, it has no column name. If you have time, you can change your code and use WITH RESULT SETS, to get a column name.
